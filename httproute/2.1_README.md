# HTTP 라우팅 (HTTP Routing)

## 개요

[HTTPRoute 리소스](https://gateway-api.sigs.k8s.io/api-types/httproute/)를 사용하면 HTTP 트래픽을 매칭하고 Kubernetes 백엔드로 전달할 수 있습니다. 이 가이드는 HTTPRoute가 호스트, 헤더, 경로 필드를 기반으로 트래픽을 매칭하고 이를 다양한 Kubernetes 서비스로 전달하는 방법을 보여줍니다.

## 트래픽 라우팅 시나리오

다음 다이어그램은 세 가지 서비스에 걸친 트래픽 흐름을 설명합니다:

* `coffee.f5bnk.com/login`으로의 트래픽은 `coffee-svc`로 전달됩니다
* `env: canary` 헤더가 있는 `tea.f5bnk.com/*`으로의 트래픽은 `coffee-svc`로 전달됩니다
* 헤더가 없는 `tea.f5bnk.com/*`으로의 트래픽은 `tea-svc`로 전달됩니다

![HTTP Routing](https://github.com/f5minions/f5-bnk/blob/kakao-poc-preparation/images/2.1_http_route.png)

점선은 이러한 라우팅 동작을 구성하기 위해 배포된 Gateway 리소스를 보여줍니다. 동일한 `prod-web` Gateway에서 라우팅 규칙을 생성하는 두 개의 HTTPRoute 리소스가 있습니다. 이는 충돌하지 않는 한 여러 Route가 Gateway에 바인딩되어 병합될 수 있음을 보여줍니다.

## 기본 구성

Gateway로부터 트래픽을 받기 위해 HTTPRoute 리소스는 연결할 부모 게이트웨이를 참조하는 `ParentRefs`로 구성되어야 합니다.

### Gateway 및 HTTPRoute 예시

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: httproute-default-gw
  namespace: web
spec:
  addresses:
  - type: "IPAddress"
    value: 192.168.48.225
  gatewayClassName: f5-gateway-class
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: "All"
      kinds:
      - kind: HTTPRoute
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: default-route
spec:
  parentRefs:
  - name: httproute-default-gw
  hostnames:
  - "*.f5bnk.com"
  rules:
  - backendRefs:
    - name: httpbin
      port: 8080
```

## 상세 라우팅 규칙

### coffee-route: 경로 기반 라우팅

다음 `coffee-route`는 `coffee.f5bnk.com`에 대한 모든 트래픽을 매칭하고 라우팅 규칙을 적용하여 올바른 백엔드로 트래픽을 전달합니다. `/login`으로 시작하는 경로만 매칭되며, 다른 경로에 대한 트래픽은 이 Route에 의해 매칭되지 않습니다.

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: coffee-route
  namespace: web
spec:
  parentRefs:
  - name: httproute-default-gw
  hostnames:
  - "coffee.f5bnk.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /login
    backendRefs:
    - name: coffee-svc
      port: 80
```

### tea-route: 헤더 기반 라우팅

`tea-route` HTTPRoute는 `tea.f5bnk.com`에 대한 트래픽을 매칭합니다. 이 호스트네임에 대한 모든 트래픽은 라우팅 규칙에 따라 평가됩니다. 가장 구체적인 매치가 우선순위를 가지므로:

* `env: canary` 헤더가 있는 트래픽은 `coffee-svc`로 전달됩니다.
* 헤더가 없거나 `canary`가 아닌 경우 `tea-svc`로 전달됩니다. 

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: tea-route
  namespace: web
spec:
  parentRefs:
  - name: httproute-default-gw
  hostnames:
  - "tea.f5bnk.com"
  rules:
  - matches:
    - headers:
      - type: Exact
        name: env
        value: canary
    backendRefs:
    - name: coffee-svc
      port: 80

    - name: tea-svc
      port: 80
```

## 주요 특징

### 호스트네임 매칭
HTTPRoute는 단일 호스트네임 집합에 대해 매칭할 수 있으며, 호스트네임은 HTTPRoute 내의 다른 매칭이 수행되기 전에 먼저 매칭됩니다.(우선 순위가 높게 먼저 매칭됨)

### Route 병합
여러 Route가 충돌하지 않는 한 Gateway에 바인딩되어 병합될 수 있습니다. 자세한 내용은 [HTTPRoute 문서](https://gateway-api.sigs.k8s.io/api-types/httproute/#merging)를 참조하세요.

### 매칭 우선순위
모든 트래픽은 라우팅 규칙에 따라 평가되며, 가장 구체적인 매치가 우선순위를 가집니다.
